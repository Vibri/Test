Intrinsic Lock (Monitor)
------------------------

public synchronized void increment() {
	counter++;
}

- every object in Java has a so-called intrinsic lock
 A thread that needs exclusive and consistent access to an object's fields
has to acquire the object's intrinsic lock before accessing them, and then 
release the intrinsic lock when it's done with them 

- because of the monitor lock: no 2 threads can execute the same synchronized method at the same time
- a thread owns the intrinsic lock between the time it has acquired the lock and released the lock
- as long as a thread owns an intrinsic lock no other thread can acquire the same lock
- the other thread will block when it attempts to acquire the lock

THE PROBLEM IS THAT EVERY OBJECT HAS A SINGLE MONITOR LOCK

- if we have 2 independent synchronized methods than the threads have to wait for each other to release the lock

Object level locking
--------------------
public synchronized void increment() {
        counter++;
}

public void increment() {
	synchronized(this) {
		counter++;
	}
}

- This is called object level locking because we get the monitor lock (intrinsic lock) 
associated with the object itself


Class level locking
---------------------
public static synchronized void increment() {
        counter++;
}

public static void increment() {
	synchronized(SomeClass.class) {
		counter++;
	}
}

- This is called class level lockig because we get the monitor lock (intrinsic lock) associated with the class

Sleep VS Wait

-  sleep  operation does not release the locks it holds while on the other hand  wait  releases the lock on the object that  wait()  is called on

THREAD POOLS
------------
When you manually create and manage lots of threads using  new Thread()  especially in large-scale or high-throughput applications, you 
run into several serious problems:
- HIGH MEMORY CONSUMPTIONS
	- Each Java  thread requires memory for its stack(typically 512 KB to 1MB per thread, depending on the JVM and platfrm)
- HIGH CPU OVERHEAD
	- Each thread has its own execution context. When many threads compete for CPU, the OS must context switch between them
- MANUAL LIFECYCLE MANAGEMENT
	- We must handle starting new threads, joining the threads if necessary, handling exceptions etc.
	
Java 
- a thread pool in Java is a collection of worker threads that are reused to execute many tasks
- instead of creating a new thread for every task (which is expensive), tasks are submitted to a pool, and threads pick up and execute them
- Java provides thread pools through the ExecutorService interface and its implementations
- threads are created once and reused for multiple tasks - it reduces overhead of frequent thread creation and destruction
- if all threads are busy, submitted tasks are queued - they are held in a BlockingQueue data structure until a thread becomes available

- newFixedThreadPool(n)
	- The thread pool is initialized lazily - threads are not created at the start but they are created on demand, up to corePoolSize
	- Once all threads are created and busy, new tasks are put into the queue and wait until a thread becomes available
	
- newCachedThreadPool()
	- the thread pool creates new threads as needed and reuses previously constructed threads when available
	- This approach automatically removes idle threads after a short timeout (after 60 seconds by default)
	- it is designed for applications that launch many short-lived asynchronous tasks
	
- newSingleThreadExecutir()
	- This creates a thread pool that uses only one worker thread to execute tasks sequentially + uses unbounded queue
	- If the thread dies (e.g., via exception), then a new thread is created automatically
	
- newScheduledThreadPool(n)
	- This creates a thread pool that can either schedule tasks to run after a delay or schedule tasks to run periodically
	- Thread starvation risk: if your core pool size is small and scheduled tasks are long-running, new scheduled tasks may be delayed
	- use it when you need tasks to run on a timer or at regular intervals
	
- SUPPORTS FUTURE NAD CALLABLE
	- you can submit task that return values using Callable and the result is wrapped in a Future object
- AUTOMATIC THREAD LIFECYCLE MANAGEMENT
	- You do not need to maually manage start() and join(). The executor handles creation, execution and shutdown of threads
- THREAD REUSE
	- Threads are created once and reused for multiple tasks. It reduces overhead of frequent thread creation and destruction
	
Decouples task submission from execution
	- You can submit tasks at any time without blocking the main thread. The main thread can continue doing other work or exit if the executor is running in the background

Improves task management and organization
	- You can queue up multiple tasks and they'll run one after another, in the order submitted. You don't have to manage thread lifecycle or manually  create or join threads

Helps avoid concurrency bugs
	- Since it quarantees tasks are run sequencially, it's safer for cases where multiple tasks share mutable state.

Joining threads
---------------
- Threads are creating dynamically
	- We want to use thread pool (e.g. ExecutorService) and we want one thread to wait for many without tracking individuals threads

- Must track each thread
	- When you use Thread.join(), you must manually track each Thread object if you want to wait for their completion - we need to store all thread references
	
Decoupling task coordination
	- CountDownLatch solves this by decoupling task coordination from thread tracking. The main thread doesn't need to know or track threads at all
	
CountDownLatch
	- is a synchronization aid in Java's java.util.concurrent package. It lets one or more threads wait until a set of operations being performed in other threads complete
	- by using a CountDownLatch we can cause a thread to block until other threads have completed a given task
	- a CountDownLatch has an internal counter, which can be decremented by calling countDown() as needed. It can be used to block a calling thread until the counter reaches zero
	- if we were performing parallelprocessing, we cound instantiate the CountDownLatch with a counter value equal to the number of threads we want to run in parallel
	- then, we could just call countDown() after each thread finishes, guaranteeing that a dependent thread calling await() will block until the worker threads are finished

Parallel Computing
-------------------

- multithreaded approach uses multiple threads but these threads are executed with the time-slicing algorithm
- MULTITHREADING IS USED TO EXECUTE INDEPENDENT TASKS WITHOUT BLOCKING OTHER TASKS
- with parallel algorithms we execute different tasks on different processors(or processor cores) simultaneously
- PARALLELIZATION IS USED TO SPEED UP A GIVEN APPLICATION

- it depends on the operating system how the threads will be executed
- if there is just a single processor core then the time-slicing algorithm executes the threads(multithreading)

Fork-Join Framework

- we do not have to bother about low level synchronizations or locks
